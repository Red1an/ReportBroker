# Report Broker

Сервис управления отчётами по конверсии просмотров/оплат товаров.

## Описание

Сервис принимает запросы на формирование отчётов через шину сообщений RabbitMQ
и предоставляет результаты через gRPC.

### Ключевые особенности

- **Дедупликация отчётов** — если несколько пользователей запрашивают
  отчёт с одинаковыми параметрами, он считается только один раз.
- **Кэширование** — результаты запросов кэшируются в Redis,
  что защищает базу данных от перегрузки при частых запросах статуса.
- **Автоматическое заполнение** — при первом запуске база данных
  автоматически заполняется тестовыми данными.

## Архитектура

Проект построен на основе Clean Architecture:
```
ReportBroker.Domain         — бизнес-объекты и интерфейсы
ReportBroker.Application    — бизнес-логика
ReportBroker.Infrastructure — БД, кэш, очередь сообщений
ReportBroker.Api            — gRPC
ReportBroker.Tests          — юнит тесты
```

### Запуск
Создать `.env` файл из шаблона.

Запустить все сервисы:
```bash
docker-compose up --build
```

## Проверка работы
### RabbitMQ UI
```
http://localhost:15672
логин: guest
пароль: guest
```

### gRPC

Метод: `report.ReportService/GetReportStatus`

Запрос:
```json
{
  "request_id": "7c9e6679-7425-40de-944b-e07fc1f90ae7"
}
```
RequestId появится в логах контейнера.

## Демонстрация дедупликации

Если отправить два сообщения с одинаковыми параметрами но разными userId.
В логах будет:
```
info: Создаём новый отчёт для продукта 550e8400...     первый запрос
info: Найден существующий отчёт для продукта 550e8400...  дедупликация
```

Оба пользователя получат один и тот же reportId.
